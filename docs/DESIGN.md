# LangChain Learning Path 设计方案

## 1. 项目定位

`LangChain Learning Path` 是一个公开的、章节化的、项目驱动的 LangChain 学习站。

它不是单纯的“LangChain 知识笔记”，也不是单纯的“代码 Demo 集合”，而是把以下三件事整合在一起：

- 学习路径：从基础概念到完整应用的渐进式路线
- 项目实战：围绕一个持续演进的主项目推进学习
- 公共传播：通过 GitHub Pages 对外公开，其他人可以直接 follow

## 2. 项目目标

- 帮助作者本人系统掌握 `LangChain / RAG / Agent / LangGraph`
- 让其他学习者按同样路径循序渐进学习
- 用统一章节模板降低读者学习成本
- 用一个贯穿式项目把碎片化知识串起来
- 第一阶段快速上线，第二阶段再补在线实验能力

## 3. 非目标

- 第一阶段不做账号系统
- 第一阶段不做多用户历史记录
- 第一阶段不追求覆盖 LangChain 所有冷门能力
- 第一阶段不做复杂社区功能

## 4. 目标用户

- 有 Python 或 JavaScript 基础，希望系统学习 LLM 应用开发的人
- 已了解 Prompt 基础，但不清楚 LangChain 整体方法论的人
- 想把 RAG、Agent、LangGraph 串成完整项目的人
- 想通过中文内容快速构建实践路径的人

## 5. 核心价值主张

- 对作者：把个人学习路径产品化、公开化、可复用化
- 对读者：不是“到处搜零散文章”，而是“按章节完成一条主线”
- 对项目本身：先做静态教学站，再逐步演进为可交互实验平台

## 6. 总体策略

采用“两阶段落地”策略。

### 阶段一：静态教学站

- 用 `VitePress` 搭建文档站
- 部署到 `GitHub Pages`
- 每章提供概念、流程图、代码、练习、总结
- 代码运行以本地示例为主

目标：

- 尽快形成完整的学习骨架
- 让站点可以稳定公开访问
- 先解决“学什么、怎么学、按什么顺序学”

### 阶段二：静态站 + 外部 LangChain 后端

- 保留 GitHub Pages 前端
- 新增独立后端承载 LangChain 运行逻辑
- 在站内提供可运行实验页
- 可加入 LangSmith Trace 展示与调试说明

目标：

- 提升交互性
- 把部分章节升级成在线实验
- 支持更多人直接体验学习成果

## 7. 产品形态

项目会由四层组成：

- 学习首页：说明项目定位、路线、如何开始
- 章节内容：按路径学习，每章固定结构
- 示例代码：每章最小可运行代码
- 综合项目：随章节逐步升级的主项目

## 8. 主学习路径设计

整个学习路径围绕“从单轮调用到完整 AI 工作流”的能力跃迁展开。

建议主线：

1. 理解 LLM 应用基础
2. 掌握 LangChain 核心抽象
3. 会写最小链式应用
4. 会搭建基础 RAG
5. 会优化 RAG 效果
6. 会做 Tool Calling 和 Agent
7. 会用 LangGraph 编排工作流
8. 能完成一个综合项目并对外展示

## 9. 贯穿式主项目

建议整站围绕一个持续演进的项目推进：`AI 学习助手`。

项目演进路径：

- 第 1 章：普通问答机器人
- 第 2 章：支持结构化输出与提示模板
- 第 3 章：支持多步骤处理链
- 第 4 章：接入课程资料形成基础 RAG
- 第 5 章：优化检索与回答质量
- 第 6 章：接入工具形成 Agent
- 第 7 章：迁移为 LangGraph 工作流
- 第 8 章：形成公开演示版本

这样可以避免每章都换一个无关的 Demo，保证读者始终在同一条业务线上学习。

## 10. 信息架构

建议站点采用如下信息架构：

- 首页
- 学习路线图
- 快速开始
- 章节目录
- 每章正文
- 示例代码入口
- 综合项目页
- FAQ
- 术语表
- 更新日志

## 11. 页面结构设计

### 首页

首页应该回答三个问题：

- 这是什么项目
- 学习顺序是什么
- 我应该从哪里开始

首页建议模块：

- Hero 区
- 项目简介
- 学习路线图
- 章节卡片
- 主项目演进图
- 为什么用这条路径
- 开始学习按钮

### 章节页

每章应采用统一结构：

- 本章目标
- 前置知识
- 核心概念
- 最小代码
- 图解流程
- 拆解说明
- 常见坑
- 练习题
- 本章总结
- 下一章导航

### 项目实战页

- 当前项目能力说明
- 本阶段架构图
- 关键代码组织
- 可运行示例说明
- 本阶段遗留问题

## 12. 内容设计原则

- 一章只完成一个明确能力跃迁
- 先最小可运行，再讲抽象和优化
- 所有代码都要有“为什么”
- 每章都要回答“你现在会了什么”
- 示例代码尽量简短，避免一开始给过大工程
- 图示优先于堆叠术语

## 13. 课程组织策略

课程按“认知复杂度递增”组织，而不是按官方文档目录机械展开。

推荐分层：

- 基础层：LLM 基础、Prompt、输出结构、LangChain 基本对象
- 应用层：链、RAG、检索、工具调用
- 编排层：Agent、LangGraph、状态管理
- 工程层：部署、日志、评估、可观测性

## 14. 技术选型

### 文档站

- `VitePress`

原因：

- Markdown 友好
- 对教程站非常合适
- 部署到 GitHub Pages 简单
- 后续可插入 Vue 组件增强页面表现

### 示例代码主语言

- `Python`

原因：

- LangChain Python 生态成熟
- 学习资料多
- RAG 和 Agent 示例更容易快速落地

### 后端

- `FastAPI`

原因：

- 结构清晰
- 文档与接口组织简单
- 适合 LangChain Python 集成

### 可选交互前端

- `React + Vite`

用途：

- 第二阶段实验页
- 可嵌入或独立部署

## 16. 章节内容资产规范

每章建议至少包含：

- 一份主文档
- 一个流程图
- 一个最小示例
- 一组练习题
- 一页“本章常见错误”

建议命名：

- `docs/chapters/chapter-01-llm-basics.md`
- `examples/chapter-01/`

## 17. 代码示例规范

- 每章一个最小可运行项目
- 文件数尽量少
- 输入、输出、依赖明确
- README 中写清运行步骤
- 不把过多工程化内容塞进前几章

每个示例建议包含：

- `README.md`
- `main.py`
- `.env.example`
- `requirements.txt`

## 18. 视觉与体验原则

项目虽然是学习站，但不应只是“普通文档页面”。

建议视觉方向：

- 清晰的学习路线图
- 强调章节进度感
- 有明显的“本章掌握能力”标签
- 架构图与流程图风格统一
- 首页要看起来像一个完整产品，而不是零散笔记

## 19. GitHub Pages 部署方案

第一阶段采用 GitHub Actions 自动部署。

流程：

1. 推送到主分支
2. GitHub Actions 构建 VitePress 静态文件
3. 自动发布到 GitHub Pages

部署前提：

- 仓库开启 Pages
- 选择 GitHub Actions 作为发布来源

## 20. 第二阶段后端架构

后端职责：

- 代理模型调用
- 运行 LangChain / RAG / Agent / LangGraph
- 封装实验 API
- 控制配额与限流
- 记录调试信息

前端职责：

- 输入实验参数
- 展示输出结果
- 展示过程说明
- 展示关键链路图

## 21. 未来实验页设计

建议按章节增量加入实验页：

- Prompt 调参实验
- Output Parser 实验
- RAG 检索实验
- Retriever 对比实验
- Tool Calling 实验
- LangGraph 状态流实验

## 22. 内容生产工作流

推荐你采用下面的内容生产节奏：

1. 先写章节目标和最小示例
2. 再写概念拆解
3. 再补流程图和练习
4. 最后统一调整术语与风格

这样不会一开始就被“大而全文档”拖慢。

## 23. 维护策略

- 用更新日志记录章节调整
- 每个季度检查一次依赖与示例
- 当 LangChain API 变化较大时优先修正示例代码
- 页面内容优先保持学习路径稳定，而不是追逐每个新特性

## 24. 风险与应对

### 风险 1：项目范围失控

应对：

- 先做 MVP
- 严格限制第一版章节数量
- 每章只讲一个能力跃迁

### 风险 2：LangChain 版本变化快

应对：

- 锁定教程对应版本
- 在每章标注适用版本
- 示例代码统一维护

### 风险 3：在线实验成本不可控

应对：

- 第二阶段再开放实验
- 先做静态课程主线
- 对实验接口加频控和配额

## 25. MVP 范围

第一版只做以下内容：

- 首页
- 路线图页
- 4 个核心章节
- 每章一个最小示例
- GitHub Pages 自动部署

优先章节：

1. LLM 应用基础
2. LangChain 核心抽象
3. 第一个链
4. RAG 入门

## 26. 里程碑

### M1：项目骨架

- 初始化仓库
- 搭建 VitePress
- 配置 GitHub Pages

### M2：课程 MVP

- 完成前 4 章
- 提供对应代码示例
- 完成首页和路线图

### M3：内容扩展

- 增加 RAG 进阶
- 增加 Agent
- 增加 LangGraph

### M4：交互升级

- 加入后端
- 上线实验页
- 加入可观测性说明

## 27. 成功标准

- 新用户能在首页看懂学习顺序
- 用户能在 1 小时内完成前两章学习
- 每章示例都能独立运行
- 站点能稳定部署到 GitHub Pages
- 综合项目能随着章节清晰演进

## 28. 下一步实施建议

建议按这个顺序继续：

1. 初始化 VitePress 站点
2. 先写首页与路线图页
3. 完成前两章内容
4. 建立 `examples/chapter-01` 和 `examples/chapter-02`
5. 配置 GitHub Pages 自动部署

下一阶段建议补齐以下文件：

- `docs/index.md`

- `docs/guide/roadmap.md`

## 30. 结论

这个项目最合理的落地方式不是一开始追求“复杂 AI 产品”，而是先把它做成一套高质量、可公开传播、可持续扩展的学习系统。

第一阶段解决“路径和内容”，第二阶段解决“交互和实验”。这样能最大化完成率，也最符合 GitHub Pages 的部署边界。
